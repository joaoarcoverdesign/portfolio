WinTeam Kiosk Device

Designing a dual-platform system that lets facility services companies deploy shared tablets for time tracking — replacing personal devices with enterprise-grade kiosk stations.

Overview

The challenge

Business Context

WinTeam is a workforce management platform built for security and facility services companies. Customers — security guards, janitors, facility workers — often operate in environments where issuing personal devices is impractical.

The Gap

Before this feature, there was no standardized way to configure a device with location, job, and language rules, manage multiple devices at scale, or ensure compliance across a fleet of kiosk devices.

How do we build an admin experience powerful enough for operations teams managing hundreds of devices, while keeping the employee-facing kiosk interface so simple that any frontline worker can use it in seconds?

Two Interconnected Surface Areas

Web platform — Admin & Supervisor tools to configure device templates and manage devices at scale. Role-based permissions, bulk operations, and compliance validation.

Kiosk app — A touch-first interface for frontline employees to clock in, clock out, and manage meal breaks without credentials or personal phones. Multilingual, offline-resilient, built for shared use.

Users & Personas

Who we're designing for

Three distinct user groups with different goals, permissions, and contexts. The design had to serve all three without compromising on security or simplicity.

Admin

Corporate Operations

Corporate/IT level, full system access across all locations and branches

Key Goals

Authorize jobs for kiosk use, manage all templates globally, remote device control, ensure compliance at scale

Supervisor

Location Manager

Branch-level manager, limited to their location's scope

Key Goals

Create/edit templates, enable kiosk mode on-device, assign devices to templates, monitor device fleet

Frontline Employee

Security Guard / Facility Worker

Warehouse, security post, facility — often multilingual, varying tech comfort

Key Goals

Clock in/out quickly, select job, manage meal breaks — with minimal friction and no training

Critical tension: Supervisors only see data within their location security scope. Admins see everything. This role-based data visibility was a core constraint throughout the design — every list, dropdown, and filter had to respect the user's permission layer.

Feature Deep Dives

Core features and design thinking

Three interconnected features — each solving a distinct problem while maintaining data consistency across the platform.

Feature 01 · Web Platform

Template Management

Operations teams need a reusable configuration unit — a "profile" they can apply to any device. A template bundles jobs, locations, language settings, time format, and geolocation rules.

Key Design Decisions

  • Location → Job dependency: When the user picks a location, the job list filters to only show jobs associated with that location. Validation prevents saving if jobs are orphaned.
  • Duplicate flow: Creates a prefilled form with "Copy of [Template Name]" — all operational fields carry over, audit fields reset.
  • Template list: Defaults to active templates. Device count chip expands inline device list without page change.

Required vs. Optional Fields

  • Required: Title, Active toggle, ≥1 location, ≥1 job, language(s), time format
  • Optional: Notes, fixed location (geolocation + range)
  • System (read-only): ID, Date Added/Changed, User Added/Changed

Feature 02 · Web Platform

Device Management

Supervisors need to monitor a fleet of devices — see their status, assigned template, last heartbeat — and make targeted edits. Admins need remote control capabilities.

Device Detail View
Read-only vs. Editable Fields

Key Design Decisions

  • Read-only vs. editable: Clear visual distinction. System fields (IMEI, software version, last contact) in read-only section with 'i' audit button.
  • Active status validation: Cannot activate a device assigned to an inactive template — toggle disabled with contextual message.
  • Remote disable (Admin): Admins can deactivate kiosk mode remotely. Only Supervisors can re-enable from device — deliberate asymmetry for accountability.
  • Bulk assignment: Multi-select → "Assign Template" → dropdown → confirmation modal → refresh list.

Feature 03 · Tablet/Mobile

Kiosk Experience

A shared device in a high-traffic, time-sensitive context. Every extra tap is friction for workers trying to start or end a shift. The interface works for employees with varying language skills and tech comfort.

Design Principles Applied

  • Large touch targets throughout
  • Minimal cognitive load — one action per step
  • Audio + visual confirmation on every punch (accessibility + noisy environments)
  • PII protection — only employee ID and job data displayed

Employee Clock-in Flow

  • Step 1: Splash screen
  • Step 2: Enter Employee ID (keypad)
  • Step 3: Clock In / Clock Out / Start Meal / End Meal (context-aware)
  • Step 4: Select Job (pre-filtered by template + location)
  • Step 5: Confirm → Success screen → back to splash

Special Features

  • Context-aware buttons: System surfaces only relevant actions based on punch state — prevents errors without error messages.
  • Multilingual: English, Spanish, French Canadian. Toggle from splash, persists through session.
  • Offline resilience: Punches stored locally, queued, auto-synced when reconnected.

Design Decisions

Key choices
and why

The thinking behind the most important design decisions — not just what was built, but the reasoning that got there.

Decision 01

Permission-First Filtering · Before/After

Permission-First Filtering

Rather than showing error states after the fact, the UI pre-filters all dropdowns and lists to the user's allowed scope. A supervisor in Omaha never sees Chicago jobs — preventing errors by never showing data a user can't act on. Maintains data integrity while reducing cognitive load.

Decision 02

Location → Job Cascading · Validation Flow

Cascading Location → Job Validation

When location changes, job list filters immediately. If a location is removed and jobs are orphaned, inline validation prevents saving — not a blocking error modal. This relationship was surfaced through real-time feedback, guiding users to fix data integrity issues before they occur.

Decision 03

Context-Aware UI · Button States

Context-Aware Kiosk Buttons

An employee who is clocked in sees "Clock Out" and "Start Meal" — not "Clock In". Prevents duplicate punches without error messages in the happy path. Reduces punch errors by making the right action obvious, not by catching mistakes after they happen.

Decision 04

Disabled with Tooltip · Discoverability Pattern

Disabled (Not Hidden) with Tooltips

"Assign to device" button is disabled (not hidden) for inactive templates — visible as a disabled state with a tooltip explaining why. Preserves discoverability while communicating constraints. Users understand the system's rules without trial-and-error.

Decision 05

Confirmation Modals · Save/Cancel Patterns

Manual Save with Confirmation Modals

All creation and edit flows use manual save with confirmation — consistent with WinTeam patterns and critical for operations context where accidental saves have real consequences. Two-option confirmations: "Save + go to list" or "Back to edit" — giving users control over next step.

Decision 06

Admin vs. Supervisor Permissions · Flow Diagram

Asymmetric Enable/Disable Permissions

Admin can disable kiosk mode remotely for security; only Supervisor can re-enable from device. This deliberate asymmetry enforces accountability — prevents unauthorized reactivation while giving Admins emergency control. Security and operational flexibility in balance.

Learnings

What I
took away

What worked well

Early Resolution of Open Questions

The requirements document surfaced many open questions. By driving answers through design decisions (not leaving them for development), the Figma deliverable became a single source of truth. This reduced implementation back-and-forth and prevented misalignment later.

What I'd push further

Metrics Dashboard Design

The metrics framework was defined but not yet instrumented. In a follow-up phase, I'd design an analytics dashboard surfacing device health and punch compliance data directly within WinTeam — closing the loop between configuration (templates) and outcomes (compliance rates).

Key insight

Templates as Connective Tissue

The dual-platform nature (web admin + kiosk tablet) required maintaining two distinct interaction paradigms while ensuring the underlying data model was consistent. The template system was the connective tissue — getting that right first unlocked everything downstream.

Constraint I navigated

Role-Based Data Visibility Everywhere

Every screen, dropdown, and filter had to respect user permissions. Supervisors see only their location; Admins see everything. This wasn't a feature — it was a fundamental constraint that shaped every interaction pattern. Permission-first filtering became the design principle that made the system work.
